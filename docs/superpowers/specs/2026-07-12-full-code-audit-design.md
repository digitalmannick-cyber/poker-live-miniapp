# 小程序全面代码检测与修复设计

## 目标

以提交 `c5babed` 为不可变基线，在 `codex/full-audit-20260712` 分支完成一次可复现、可回退、按风险分层的代码检测与修复。优先发现会造成数据丢失、统计错误、云端不同步、牌局错误保存或核心页面不可用的问题；不借机重构无关代码，也不改变已确认的产品语义。

## 基线与回退

- GitHub 备份标签：`backup/pre-full-audit-2026-07-12`
- 基线提交：`c5babed`
- 修复分支：`codex/full-audit-20260712`
- 检测开始前和最终交付时，工作区必须能够由 Git 精确描述。
- 每类独立缺陷使用独立提交，禁止把无关格式化或资源更新混入修复提交。

## 检测策略

采用“证据先于修改”的风险分层流程：

1. 建立基线：记录仓库状态、Node 语法检查结果、全部现有测试结果和失败清单。
2. 静态检测：检查语法、失效路由和组件引用、明显死代码、硬编码业务种子、危险数值转换、异步错误吞噬以及本地/云端字段不一致。
3. 核心链路审计：按数据所有权和用户损失风险检查持久化、同步、录入、复盘、统计、设置与导入导出。
4. 问题复现：每个准备修复的问题必须具备失败测试、确定性复现步骤或可定位的静态证据。
5. 最小修复：只修改根因相关代码；涉及产品语义、历史数据迁移或云端破坏性写入的候选问题先报告，不擅自实施。
6. 分层回归：先运行聚焦测试，再运行全量测试，最后用微信开发者工具对真实工作区自动预览。

## 风险分级与处理规则

### P0：数据安全或不可恢复故障

包括数据被覆盖/删除、跨用户数据串用、导入导出不可恢复、云同步造成持久数据损坏。确认后立即停止扩大检测范围，先补测试并修复；任何需要操作真实云数据的验证均不得自动执行。

### P1：核心业务结果错误

包括牌局输赢、筹码、时长、统计、All-in EV、牌局归属或会话状态错误，以及核心录入/保存/复盘流程不可用。必须有复现证据并优先修复。

### P2：功能或兼容性回归

包括页面入口失效、字段显示错误、异常状态未处理、旧数据兼容失败、明显性能退化。修复应保持现有数据模型和路由。

### P3：低风险质量问题

包括不可达代码、重复逻辑、非阻断样式问题和缺少保护性测试。只有在改动局部、收益明确且不会扩大回归面时才修复；否则进入报告。

以下情况只记录、不直接修改：需求含义不明确；需要推断用户偏好；可能改变历史统计口径；需要新增前端语义解析或回退；需要批量修改真实云端数据；无法用测试或稳定步骤证明问题。

## 审计范围

### 1. 数据与持久化

检查 `utils/store.js`、`services/data-service.js`、`services/cloud-repo.js`、`services/cloud-data-api.js` 与 `cloudfunctions/poker_data/`。重点验证本地和云端字段契约、默认值、数值合法性、迁移幂等性、缓存失效、用户隔离、大备份拆分及失败后的数据保留。

### 2. 牌局录入与会话

检查 `pages/hand-record/`、`pages/hand-ledger-input/`、`pages/session-list/`、`pages/session-detail/` 及相关工具模块。重点验证输赢为有限数值、单一活跃会话、牌局归属、时间排序、全量/快速录入元数据、保存失败提示和重复提交保护。

### 3. 复盘与 Agent 边界

检查 `pages/review-list/`、`pages/hand-detail/`、`components/agent-chat/`、`services/voice-service.js` 与 `cloudfunctions/poker_review/`。Agent/backend 的结构化字段是语义真源；小程序只展示、允许用户确认编辑并持久化，不新增 action line、位置、底池、牌面、标签、equity 或 All-in EV 的本地解析与兜底。

### 4. 统计与资金

检查 `pages/stats/`、`utils/stats-analytics.js`、统计缓存和云端摘要。重点验证本金初始值、输赢汇总、澳门规则、showdown EV、空数据、历史数据、缓存与云端紧凑载荷的一致性。

### 5. 设置、提醒、玩家笔记和导入导出

检查 `pages/profile/`、AI 提醒组件、玩家笔记、PBT 导入和 Agent/牌局导出。重点验证设置持久化、账号边界、资源缓存、失败回滚、导出契约和只读导出行为。

### 6. 配置、资源与可运行性

检查 `app.json`、组件 JSON、`project.config.json`、云函数依赖、资源引用和打包边界。验证所有注册页面/组件存在，关键 JSON 可解析，JavaScript 可加载，预览包不包含无关根目录产物。

## 测试矩阵

### 自动化基线

- 使用当前 Node 运行时逐个执行 `tests/*.test.js`，保存通过、失败和超时清单。
- 对仓库业务 JavaScript 运行 `node --check`；排除依赖目录和明确仅供微信运行时动态注入的非标准源。
- 解析应用、页面、组件和项目 JSON，并核对 `app.json` 中注册路径对应的文件集合。
- 聚焦测试必须在每个修复前失败、修复后通过；若现有测试错误地固化了缺陷，应在同一提交中说明并修正断言。

### 回归层级

1. 修改文件对应的聚焦测试。
2. 相邻模块测试，例如 store 修复同时运行 data-service、cloud stats、导入和统计测试。
3. 全部 `tests/*.test.js`。
4. 微信开发者工具 CLI 对真实工作区执行 auto-preview，记录结果和包体大小；预览与上传分离，不自动上传开发版本。

### 手工烟雾检查

预览中至少覆盖启动、会话列表、新建/继续会话、快速录入、完整录入、复盘列表、牌局详情、统计、玩家笔记和设置入口。只使用演示或本地数据验证；不会对真实云数据执行删除、迁移或批量写入。

## 缺陷记录格式

每个发现记录：编号、级别、影响面、复现证据、根因、修复文件、增加或修改的测试、回归结果、剩余风险。无法确认的问题放入“待确认观察项”，不混入已修复统计。

## 完成标准

- P0、P1 已确认问题全部修复，或明确列出因权限/产品决策而阻塞的项目。
- P2 已确认问题修复，无法安全修复的有证据和建议。
- 全量测试无新增失败；基线已有失败须说明是否修复及原因。
- 所有业务修复都有复现测试或等价确定性证据。
- 微信开发者工具真实工作区 auto-preview 成功；若环境阻塞，保留原始错误并给出可复现命令。
- `git diff` 不含无关代码，工作分支可与备份标签直接比较并随时回退。
- 最终报告明确区分“已修复”“测试已覆盖”“仅观察到”“未验证”。

## 非目标

- 不进行全项目风格统一或大规模架构重写。
- 不新增产品功能，不改变页面信息架构和视觉方向。
- 不把 Agent/backend 语义提取复制到小程序端。
- 不自动上传开发版本，不部署云函数，不修改真实云数据。
- 不以删除失败测试来获得全绿结果。
