## 1. 目标与边界

广场分享只包含云端生成的结构化牌谱：金额统一为 BB、非 Hero 玩家使用固定匿名名称、没有用户自填标题、正文或图片。因此结构化牌谱本体不做通用内容检测，也不为牌谱或评论增加逐条举报入口。用户实际输入的广场文字评论与回复在写入前接入微信服务端文本安全检测。

广场仍允许所有已登录社交用户发表文字、Emoji 和内置扑克贴纸评论。为了在出现广告、辱骂或隐私泄露评论时不必直接修改数据库，第一版增加一项最小应急能力：经过云端 OpenID 白名单鉴权的管理员可以软删除任意评论，并留下不可由客户端篡改的操作审计。

本设计不改变以下既有产品决定：

1. 评论作者可以删除自己的评论。
2. 手牌发布者不能仅因自己是发布者而删除他人评论。
3. 动态列表继续只显示点赞、评论两个小图标。
4. 仅增加广场文字评论与回复的自动检测，不增加逐条举报、屏蔽、申诉或完整审核工作台。
5. 用户仍可通过个人页现有“反馈与建议”联系客服。

## 2. 管理员定义

### 2.1 权威身份

云函数环境变量使用以下格式：

```text
SOCIAL_ADMIN_OPENIDS=openid_a,openid_b
```

云函数启动时把该值按英文逗号拆分、去除首尾空白、过滤空值并去重，形成只存在于服务端内存中的管理员 OpenID 集合。环境变量缺失、为空或格式无效时，系统视为没有管理员，所有管理员动作都失败关闭。

管理员身份只能由当前云函数请求的微信上下文 OpenID 与该集合匹配确定。以下字段一律不能授予管理员权限：

- 昵称或头像；
- 私有玩家 ID；
- 社交用户 ID；
- 客户端传入的 `isAdmin` 或能力字段；
- 本地缓存或页面显示状态。

### 2.2 客户端能力暴露

`get_hand_share` 在原有严格 DTO 上增加：

```js
canModerateComments: boolean
```

该值只表示当前请求账号能否显示管理员删除入口，不返回角色名、管理员数量或任何管理员 OpenID。真正执行删除时，云函数必须再次根据当前微信上下文 OpenID 校验，不能把这个布尔值当作授权凭据。

## 3. 管理员删除流程

### 3.1 页面交互

当 `canModerateComments === true` 且评论未删除时，手牌详情页在评论现有“回复/删除”操作区显示一个管理员“移除”文字操作。点击后使用原生 ActionSheet 选择固定原因：

- 垃圾广告；
- 骚扰或攻击；
- 泄露隐私；
- 违法或欺诈；
- 其他违规。

提交期间锁定该评论的管理员操作，成功后立即使用服务端返回的评论 DTO 更新页面；失败显示“移除失败，请重试”。页面卸载、账号切换或分享失效后，旧请求不得回填新账号页面。

普通评论作者仍看到原有“删除”。管理员同时也是评论作者时，页面保留作者“删除”，不重复显示管理员“移除”。

### 3.2 服务接口

客户端服务增加：

```js
adminDeleteComment({ commentId, reason, clientMutationId })
```

对应云函数 action：

```text
admin_delete_comment
```

允许的 `reason` 固定为：

```text
spam | abuse | privacy | illegal | other
```

接口必须要求非空、最长 128 字符的 `clientMutationId`，并复用现有社交幂等框架。同一管理员、同一 mutation ID、同一输入重复调用返回同一公开结果；相同 mutation ID 对应不同评论或原因时返回 `MUTATION_CONFLICT`。

### 3.3 云端事务

事务顺序固定为：

1. 依据当前请求 OpenID 再次校验管理员白名单。
2. 读取评论并验证严格记录形状。
3. 读取对应分享，验证评论确实属于该分享。
4. 若评论仍有效，写入软删除状态并把分享 `commentCount` 安全减一。
5. 写入管理员操作审计。
6. 完成社交 mutation 记录并返回隐藏正文后的评论 DTO 与最新 `commentCount`。

评论记录新增的内部字段为：

```js
{
  deleted: true,
  deletedAt: number,
  updatedAt: number,
  deletionKind: 'admin',
  moderationReason: 'spam' | 'abuse' | 'privacy' | 'illegal' | 'other'
}
```

正文和原作者快照保留在禁止客户端直读的记录中，用于必要的运营核查；`toCommentDto` 不返回正文、原因或管理员信息，只返回现有 deleted DTO，并把展示文案改为“该评论已被管理员移除”。作者自行删除时使用 `deletionKind: 'author'`，继续显示原有“该评论已删除”。对没有 `deletionKind` 的旧删除记录按作者删除兼容。

已由作者删除的评论不再产生管理员处置；已由管理员删除的评论重复调用返回当前删除结果，不能再次扣减评论数或重复写入审计。

## 4. 操作审计

新增私有集合：

```text
social_moderation_audits
```

每条记录包含：

```js
{
  action: 'admin_delete_comment',
  targetType: 'comment',
  commentId: string,
  shareId: string,
  targetAuthorId: string,
  moderatorId: string,
  reason: 'spam' | 'abuse' | 'privacy' | 'illegal' | 'other',
  clientMutationId: string,
  targetRedacted: boolean,
  moderatorRedacted: boolean,
  createdAt: number
}
```

`moderatorId` 使用当前管理员的社交用户 ID，不保存原始 OpenID。集合安全规则固定为客户端不可读、不可写。记录 ID 由管理员、action、评论 ID 和 mutation ID 生成不可猜测哈希，避免重复及跨管理员冲突。

账号清除时审计记录不作为业务数据恢复来源，也不原样保留账号标识：

- 被处置评论作者清除账号时，清空审计中的 `commentId`、`shareId`、`targetAuthorId` 和 `clientMutationId`，设置 `targetRedacted: true`。
- 管理员清除账号时，清空 `moderatorId`，设置 `moderatorRedacted: true`。
- 同一账号同时命中两个身份时同时执行两类脱敏。
- 脱敏后只保留 action、固定 reason 和 createdAt，不能通过审计恢复评论、分享、社交资料或账号关系。

第一版不提供审计列表页面；运营核查通过 CloudBase 控制台进行。后续完整审核工作台可以读取同一集合，不需要改变评论记录语义。

## 5. 权限与错误

新增公开错误码：

```text
INVALID_MODERATION_REASON
```

权限失败继续使用现有 `FORBIDDEN`，不向调用者区分“不是管理员”“白名单未配置”或“管理员已被移除”，避免泄露管理员配置。

管理员仍必须是有效、未清除中的社交账号。分享撤回或好友关系变化不会赋予普通用户删除权限；管理员可以处置仍存在的评论记录，即使对应分享已经撤回，但此时公开结果不返回评论计数。

## 6. 数据库与部署

部署需要同步完成：

1. 先部署数据库安全规则，显式拒绝客户端访问 `social_moderation_audits`。
2. 再部署审计集合索引：账号清除只使用 `targetAuthorId` 与 `moderatorId` 两种查询索引，不增加其他未使用索引。
3. 随后部署包含审计写入与账号清除新阶段的 `poker_social` 云函数；清除失败时整条账号清除流程必须失败关闭并允许同一 mutation 重试。
4. 最后为 `poker_social` 配置 `SOCIAL_ADMIN_OPENIDS`，初始值只包含确认过的管理员微信 OpenID。未配置期间管理员能力保持失败关闭。
5. 云函数发布后使用一个管理员账号和一个普通账号完成真实权限验证。

管理员名单变化属于云端配置变更，不需要发布小程序前端，但需要重新部署或更新云函数环境变量。不得把管理员 OpenID 写进代码仓库、测试夹具、日志、客户端缓存或 DTO。

## 7. 测试与验收

### 7.1 云函数测试

必须覆盖：

1. 白名单解析去空白、去空值、去重；未配置时失败关闭。
2. 普通用户伪造 action、客户端能力字段或其他身份字段仍返回 `FORBIDDEN`。
3. 管理员删除有效评论后，评论软删除、计数减一、审计写入一次。
4. 作者已删除评论不能被管理员重复处置。
5. 管理员重复 mutation 返回相同结果且不重复扣数或写审计。
6. 相同 mutation 对不同输入返回 `MUTATION_CONFLICT`。
7. 删除不存在或畸形评论失败关闭。
8. 分享已撤回时可以软删除残留评论，但不泄露分享或计数。
9. DTO 不返回原因、管理员 ID、OpenID、原评论正文或审计记录。
10. 新审计集合拒绝客户端读写并进入部署范围契约。
11. 目标作者或管理员清除账号时，审计身份和目标标识按角色脱敏，且脱敏记录不能恢复任何业务数据。

### 7.2 页面测试

必须覆盖：

1. 普通用户看不到管理员“移除”。
2. 管理员看到未删除的他人评论“移除”，自己的评论只显示原作者“删除”。
3. 取消原因选择不调用服务。
4. 连续点击同一评论只发出一个请求，并复用稳定 mutation ID。
5. 成功后显示“该评论已被管理员移除”并更新计数。
6. 失败后解锁且允许使用同一 mutation 重试。
7. 页面卸载、账号切换及 ABA 切换后旧请求不能回填。
8. 动态卡片仍只有点赞、评论两个小图标。

### 7.3 回归与真实验证

完成定向测试后，必须运行全部社交测试、全部仓库串行测试和微信开发者工具干净包 `auto-preview`。真实环境至少使用两个账号验证：普通账号评论，管理员账号删除，普通账号重新打开详情确认正文不可见且评论数一致；再由普通账号伪造管理员 action，确认云端拒绝且没有任何数据变化。

## 8. 可拓展功能

以下能力不进入本次实现：

- 图片、好友范围评论等更广内容类型的自动检测；公开昵称已经使用资料场景检测；
- 牌谱举报或逐条评论举报；
- 举报队列和完整审核工作台；
- 用户申诉；
- 用户屏蔽和仅本地隐藏评论；
- 多级管理员角色与在线授权管理。

未来增加这些能力时，继续以云函数鉴权和私有审计集合为权威，不得把管理员权限下放给客户端。
