# 微信好友系统融合需求文档

## 文档状态

- 日期：2026-07-19
- 状态：已确认
- 适用项目：`poker-live-miniapp`
- 交付策略：内部按两个开发里程碑实施，全部验收通过后一次性发布完整功能

## 1. 背景

现有小程序已经具备微信账号登录、场次记录、手牌复盘、玩家库、统计和个人资料能力，但所有业务数据均按当前账号隔离，用户之间不存在好友关系或跨账号内容访问。

本功能在不暴露盈亏和私人牌局数据的前提下，增加小程序内双向好友关系，使用户能够查看好友的非财务投入数据、讨论主动发布的单手牌，并在明确授权下分享玩家库中的玩家名片。单手牌可以发布到公开广场、全部好友或指定好友；所有范围都使用同一套 BB 化与匿名化规则。

普通微信小程序不能读取当前用户的微信好友列表，也不能判断两个登录用户是否互为微信好友。因此本需求中的“好友”均指通过微信邀请卡片或个人二维码发起、由对方确认后建立的“小程序好友”，不是自动同步的微信通讯录关系。

## 2. 产品定位

好友系统定位为“好友关系 + 隐私安全的手牌讨论”。好友资料、统计、排行榜和玩家名片仍属于封闭好友关系；只有用户明确选择“广场”发布的脱敏手牌允许已登录的非好友查看和互动。广场不开放用户私有资料、场次、玩家库或未主动发布的手牌。

### 2.1 目标

1. 用户可以通过微信邀请卡片或个人二维码建立双向小程序好友关系。
2. 好友可以在隐私规则允许时查看彼此的累计牌局时长、记录手牌数和称号。
3. 用户可以主动向广场、全部好友或指定好友发布一手经过 BB 化和匿名化处理的手牌。
4. 当前发布范围内的已登录用户可以对分享手牌评论、回复、发送 Emoji 或内置扑克贴纸，并点赞。
5. 用户可以把玩家库中的一张完整玩家名片分享给一位指定好友；接收方预览后手动导入自己的玩家库。
6. 好友可以在不展示盈亏的排行榜中按有效牌局时长比较本周、本月和累计排名。
7. 所有社交消息通过小程序内消息中心统一承载。
8. 社交功能故障不得阻塞现有场次、记牌、玩家库、复盘和统计流程。

### 2.2 成功标准

- 只有广场分享允许非好友读取；好友范围和指定范围必须严格鉴权。
- 好友页面和排行榜不返回、不展示、也不依赖任何盈亏字段。
- 分享手牌不包含真实金额、真实对手身份或玩家库私人字段。
- 玩家名片导入不会破坏接收方已有的对战手牌关联。
- 解除好友或撤销权限后，旧页面刷新立即失去访问能力。
- 用户能够从邀请到讨论手牌、从名片预览到导入完成闭环。

## 3. 术语

- **小程序好友**：通过本小程序发起并双向确认的好友关系。
- **分享手牌**：用户主动发布到广场、全部好友或指定好友的独立脱敏手牌快照。
- **分享快照**：发布当时从私有数据提取并脱敏生成的不可自动变化副本。
- **玩家名片**：从玩家库提取的头像、名称、玩家类型、Leak 标签和 Note 快照。
- **全部好友**：发布确认时与发布者保持有效好友关系且未被隐私规则排除的全部好友。
- **指定好友**：发布者在一次分享中明确选择的一个或多个有效好友。
- **广场**：所有已登录小程序用户均可查看和互动的公开手牌信息流；不等于公开用户资料或开放私有数据库读取。
- **有效牌局时长**：已经结束、起止时间有效且能够计算出正数分钟的牌局时长。

## 4. 信息架构与入口

现有底部五个导航保持不变，“玩家”仍使用原名称。

```text
底部导航：玩家
├─ 好友
│  ├─ 动态
│  ├─ 好友列表
│  └─ 排行榜
└─ 玩家库
```

### 4.1 玩家页

- 页面一级切换为“好友 / 玩家库”。
- “好友”下提供“动态、好友、排行榜”三个入口，默认进入“动态”。
- “动态”是单一信息流，不再拆分“好友动态”和“广场动态”页面，也不设置二级范围切换。
- 当前用户有权查看的“广场、全部好友、指定好友”内容统一按发布时间倒序混排，并以范围标签说明可见对象。
- “玩家库”承载现有玩家库能力，不改变现有玩家档案和对战手牌语义。
- 页面右上角增加消息中心入口并显示未读数量。
- 底部“玩家”tab 在存在未读社交消息时显示红点，不显示未读数字。

### 4.2 我的页

“我的”页面增加“隐私与分享”设置入口，至少包含：

- 好友统计数据可见开关，默认开启。
- 手牌分享默认范围说明，默认“全部好友”，可切换为“广场”或“指定好友”。
- 分享手牌固定 BB 化、匿名化规则说明；第一版不提供关闭选项。

### 4.3 原业务入口

- 手牌详情增加“分享给好友”。
- 玩家详情增加“分享玩家名片”。
- 现有手牌导出、私人评论、玩家 Note 和场次记录入口保持原语义，不自动转为社交分享。

## 5. 好友关系

### 5.1 添加好友

第一版提供两种方式：

1. 微信邀请卡片：用户把小程序邀请卡片发送到微信会话。
2. 个人好友二维码：用户当面展示二维码，由对方使用微信扫码打开。

邀请卡片或二维码只标识邀请人并引导接收方发送好友申请，不能自动建立好友关系。卡片被转发后，新的打开者也只能发出申请。

第一版不开放全站昵称搜索，不通过现有账号 ID 建设公开用户目录。

### 5.2 好友申请状态

好友关系至少包含以下状态：

- `pending`：申请已发出，等待接收方处理。
- `accepted`：双方已建立好友关系。
- `rejected`：接收方拒绝申请。
- `removed`：任一方解除已建立的好友关系。

同一对用户之间只能存在一条有效关系记录。双方同时发起申请时，系统合并为一条待处理关系，不产生重复好友。

### 5.3 申请冷却

- 申请被拒绝后，双方 7 天内不能再次发起好友申请。
- 好友关系被解除后，双方 7 天内不能再次发起好友申请。
- 冷却期结束后可以重新申请，仍需对方确认。

### 5.4 解除好友

- 任一方可以解除好友关系，不需要对方确认。
- 解除后双方立即失去好友主页统计、排行榜以及全部好友/指定好友分享的访问权。
- 双方不能继续对旧的好友范围分享评论、回复或点赞；广场分享仍按公开权限处理。
- 历史评论和点赞保留在原分享记录中，不因解除好友而物理删除。
- 已经导入接收方玩家库的玩家名片副本不受影响。
- 第一版不提供内容举报、用户屏蔽和“仅对我隐藏评论”；管理员依据服务端 OpenID 白名单执行的评论处置不属于好友关系权限。

## 6. 好友主页与统计隐私

### 6.1 好友主页可见内容

- 好友首次建立时读取对方当时在小程序中选定的头像和昵称；对方可以使用微信头像、微信昵称，也可以使用自己设置的头像和昵称。
- 当前用户可以仅在自己这一侧修改该好友的显示头像和备注名称，不影响对方账号资料，也不影响其他好友看到的资料。
- 好友详情复用现有玩家详情页面和字段结构，包括玩家类型、Leak 标签、Note 和对战手牌；这些字段以及本地头像、备注名称仅当前用户可见和编辑。
- 好友详情在玩家库基础信息之上增加当前称号、累计有效牌局时长和累计记录手牌数；称号和统计来自云端好友资料，不能由查看者本地编辑。
- 当前称号
- 累计有效牌局时长
- 累计记录手牌数
- 最近主动发布且当前用户有权访问的分享手牌
- 成为好友的日期
- 解除好友入口

### 6.2 禁止展示内容

- 累计盈亏和单场盈亏
- 小时盈利、胜率和资金曲线
- 买入、带走和资金记录
- 具体场次、地点、盲注和货币单位
- 好友自己的玩家库及私人玩家 Note；当前用户为该好友建立的本地玩家类型、Leak、Note 和对战手牌仍只属于当前用户
- 好友关系网、共同好友或好友数量
- 未主动发布的手牌

### 6.3 好友统计可见开关

- 开关默认开启。
- 关闭后好友关系继续保留。
- 好友主页仅显示头像、昵称和“对方已隐藏统计数据”。
- 关闭后用户从本周、本月和累计排行榜中移除。
- 关闭统计不影响用户主动发布手牌或分享玩家名片。

## 7. 分享手牌发布

### 7.1 发布范围

- 默认范围为“全部好友”。
- 用户可以切换为“广场”；发布后所有已登录小程序用户均可查看、评论和点赞。
- 用户可以切换为“指定好友”并选择一个或多个有效好友。
- 三种范围互斥，一次发布只能选择一种。
- 发布确认页必须明确显示“广场公开”“全部好友可见”或“指定 N 位好友可见”。
- 选择“广场”时增加二次提示：“公开发布后，非好友也可查看和评论；手牌仍会强制 BB 化并匿名化其他玩家。”
- 修改已发布手牌的范围需要再次确认；从私域改为广场时必须再次展示公开发布提示。

### 7.2 分享快照

分享采用“私有原手牌 → 独立分享快照”，不得把私有手牌文档直接开放给其他账号。

快照必须包含或允许包含：

- 发布者好友身份和头像，用于帖子头部展示。
- 发布者座位，统一显示为 `Hero`。
- Hero 手牌。
- 公共牌。
- 座位位置。
- 行动线。
- 底池及行动金额的 BB 数。
- 摊牌结果。
- 原始记录中明确摊牌的对手底牌。

快照必须排除：

- 真实金额、货币单位和真实级别金额。
- `profit`、单手资金变化、累计盈亏或场次结果。
- 买入、带走和资金曲线数据。
- 场次名称、具体地点和私人场次备注。
- 对手真实姓名、头像及玩家库关联。
- 玩家类型、Leak 标签和 Note。
- AI 私人分析和私人手牌评论。
- 未摊牌或由系统推测的对手底牌。

### 7.3 BB 化

- 第一版所有分享手牌统一使用 BB。
- 不提供切换为真实金额的选项。
- 所有下注、加注、跟注、底池和有效筹码数值使用一致的 BB 口径。
- 当原始手牌缺少可可靠换算 BB 的盲注信息时，禁止发布并提示用户先补全手牌级别。

### 7.4 匿名代号

- 非 Hero 玩家强制匿名。
- 第一版使用原创怪盗风代号池：夜鸦、赤狐、黑猫、银狼、幻蝶、灰隼、绯蛇、白鲸。
- 每条分享中按稳定座位顺序分配代号。
- 同一玩家在该手牌所有街和评论引用中保持同一代号。
- 不使用 Persona 5 官方角色名称。

### 7.5 快照一致性

- 分享快照在发布后不随原始手牌编辑而自动改变。
- 需要修订时，发布者先撤回旧分享，再重新发布新快照。
- 评论和点赞绑定分享快照，不绑定可变化的原始手牌。
- 发布者修改可见范围后，所有读取立即按新范围重新鉴权。

### 7.6 撤回与删除

- 发布者可以撤回整条分享手牌。
- 发布者不能删除或隐藏其他人的评论。
- 撤回后手牌、评论和点赞整体停止展示。
- 发布者删除原始手牌时，系统自动撤回该原手牌产生的全部分享。
- 撤回和关联互动采用软删除或不可见状态，以保留一致性和审计上下文。

## 8. 评论、回复、贴纸与点赞

### 8.1 评论能力

- 支持文字和手机键盘 Emoji。
- 支持应用内置扑克表情贴纸。
- 支持回复某条评论，最多一层回复。
- 评论数据保持扁平结构，回复通过父评论标识关联。
- 单条文字评论最长 300 个字符。
- 单条评论可以是文字、文字加 Emoji，或者一张内置贴纸。
- 第一版不允许上传任意图片、GIF、语音或外部贴纸地址。

### 8.2 删除规则

- 评论者可以删除自己的评论。
- 发布者不能删除其他人的评论。
- 自己的评论删除后显示“该评论已删除”，已有回复保留上下文。
- 发布者不希望继续展示讨论时，只能撤回整条分享手牌。

### 8.3 点赞

- 当前发布范围内的已登录用户可以点赞和取消点赞。
- 动态卡片以小图标加数量展示点赞和评论入口，不使用两列大按钮；图标视觉尺寸约 16px，实际点击热区不得小于 36px。
- 点赞激活后图标变为红色并提供轻微状态动效，评论图标点击后打开评论面板。
- 同一用户对同一分享最多存在一个有效点赞。
- 重复提交不会增加重复点赞记录。
- 第一版不支持给评论点赞。

### 8.4 关系变化

- 解除好友后双方不能继续读取或互动全部好友/指定好友范围的分享；广场分享不受好友关系变化影响。
- 历史评论和点赞仍保存在分享记录中。
- 评论保存发送当时的昵称和头像快照，用于保持历史上下文。
- 重新成为好友后，旧分享是否可见仍由该分享当前可见范围决定。

## 9. 玩家名片分享与导入

### 9.1 分享内容

玩家名片包含：

- 头像
- 玩家名称
- 玩家类型
- Leak 标签
- 完整 Note

玩家名片不包含：

- 历史兼容字段 `alias`
- 对战手牌及 `battleHandIds` / `linkedHandIds`
- 该玩家的输赢、统计和场次数据
- 分享者与该玩家的关系数据
- 原始数据库 ID、云文件内部字段
- 创建时间、更新时间等系统字段

### 9.2 分享对象

- 一次操作只能选择一张玩家名片。
- 一张玩家名片一次只能分享给一位指定好友。
- 不提供默认全部好友或多选好友。
- 只有被指定好友能够打开该名片分享。

### 9.3 分享与预览流程

1. 分享者进入玩家详情并点击“分享玩家名片”。
2. 页面完整展示将要分享的头像、名称、类型、Leak 标签和 Note。
3. 分享者选择一位有效好友并确认。
4. 系统生成独立名片快照和目标好友访问权限。
5. 第一版由接收方从消息中心打开受限预览；微信会话中的系统分享卡片作为后续引流能力单独评估，不作为私密名片送达链路。
6. 接收方查看完整预览后手动选择保存。

### 9.4 有效期和所有权

- 未导入的玩家名片分享 7 天后失效。
- 分享者可以在有效期内撤回，撤回后尚未导入的接收方不能继续访问。
- 名片导入后成为接收方自己的独立副本。
- 分享者后续修改原玩家不会自动同步。
- 分享者不能远程修改或删除已经导入的副本。
- 解除好友关系不影响已经导入的副本。

### 9.5 重复检测和覆盖

- 标准化后的玩家名称完全相同时提示“疑似已存在”。
- 不根据历史 `alias` 字段判断重复。
- 接收方选择“覆盖已有玩家”或“新建玩家”。
- 不提供字段逐项合并。
- 覆盖时替换头像、名称、玩家类型、Leak 标签和 Note。
- 覆盖时保留接收方已有玩家 ID、对战手牌关联和系统字段。
- 覆盖前明确列出会替换的名片内容和会保留的本地关系。

## 10. 排行榜

### 10.1 排名范围

- 页面名称为“排行榜”。
- 默认显示“本周”。
- 支持“本周 / 本月 / 累计”切换。
- 本周按北京时间周一 00:00 至下周一 00:00 计算。
- 本月按北京时间自然月计算。

### 10.2 排名口径

- 只按有效牌局时长排序。
- 只统计已经结束且时长为正数的牌局。
- 进行中的场次不进入排行榜。
- 内部使用分钟数比较，页面转换为小时显示。
- 时长完全相同的用户使用并列名次。
- 页面可以同时显示记录手牌数，但不能切换为手牌数排名。
- 不使用盈亏、胜率或手牌数打破平局。
- 自己的排名固定展示，即使不在当前首屏。
- 榜单主体只显示 Top 10；自己未进入 Top 10 时，在榜单底部固定显示自己的实际名次。
- 关闭好友统计可见的用户不进入任何排行榜。
- 第一版不提供排行榜奖励。

## 11. 消息中心

### 11.1 入口

- “玩家”页右上角显示消息图标和未读数量。
- 底部“玩家”tab 存在未读消息时显示红点。
- 消息中心支持“一键全部已读”。

### 11.2 消息类型

- 收到好友申请，可直接接受或拒绝。
- 好友申请通过。
- 被指定接收一手牌。
- 其他用户评论或回复了自己的手牌。
- 其他用户点赞了自己的手牌。
- 收到指定玩家名片。

连续点赞可以聚合为“某好友等 N 位好友赞了你的手牌”。玩家名片即将过期不额外发送提醒。

### 11.3 失效目标

消息目标已撤回、已过期或权限已取消时，点击后显示明确的“内容已不可访问”，不能进入空白页或抛出未处理错误。好友关系解除后，好友范围内容失效；广场内容仍按其当前公开状态处理。

第一版只做小程序内消息中心，不接微信订阅消息。

## 12. 在线、错误与内容安全

### 12.1 在线边界

- 好友关系、好友主页、排行榜、广场、消息、评论、点赞和分享均为在线功能。
- 云端不可用时显示“好友功能暂时不可用”。
- 已缓存内容最多以只读方式展示。
- 离线状态不能评论、点赞、接受好友申请或发布分享。
- 第一版不维护离线社交写入队列。
- 网络恢复后由用户刷新或重新进入页面获取最新状态。

### 12.2 权限校验

- 所有跨账号读取和写入都必须由云端重新校验登录状态、好友关系、隐私设置、分享范围、内容状态和有效期。广场读取不要求好友关系，但仍要求有效登录和内容可见状态。
- 前端隐藏按钮不能代替权限控制。
- 客户端不能获得其他用户的 OpenID、原始数据库 ID或完整私有文档。
- 分享令牌必须不可猜测，并与分享目标和有效期绑定。
- 重复接受申请、重复点赞、重复发布和重复导入必须具备幂等保护。

### 12.3 第一阶段内容边界

- 广场允许所有已登录用户浏览、点赞、评论和回复；广场中的文字评论与回复在写入前必须通过微信服务端文本安全检测。
- 检测结果为 `risky`、`review`，或检测接口不可用、返回结果异常时，评论均不得写入；客户端分别提示用户修改内容或稍后重试。
- 检测仅覆盖用户实际输入的广场文字。好友/指定好友范围的评论、内置扑克贴纸、牌谱快照、玩家 Note、Leak 和好友资料不送检。
- 评论仍执行长度、数据格式和频率限制；内置贴纸只接受预定义贴纸 ID。
- 广场分享不提供自由输入标题，不展示可反向识别真实牌局的地点、场次名或真实玩家名，因此结构化牌谱本体不做通用内容检测。
- 第一阶段不提供帖子或评论举报入口；逐条举报、完整审核工作台和用户申诉列入可拓展功能。
- 第一阶段保留管理员软删除评论能力作为漏检后的人工兜底；用户仍可通过现有“反馈与建议”联系客服。

### 12.4 管理员评论处置

- 管理员由云函数环境变量 `SOCIAL_ADMIN_OPENIDS` 中的微信 OpenID 白名单定义，允许配置一个或多个账号。
- 客户端不得获得管理员 OpenID 列表；页面只接收当前账号是否具备 `canModerateComments` 能力的布尔值。
- 管理员删除评论时，云函数必须重新以当前微信上下文 OpenID 校验白名单，不能信任客户端能力标记、昵称、玩家 ID 或社交用户 ID。
- 管理员可以软删除任意仍存在的评论；评论作者原有的自删能力不变，手牌发布者仍不能因发布者身份删除他人评论。
- 管理员删除必须幂等，原评论正文保留在禁止客户端直读的云端记录中，对客户端统一显示“该评论已被管理员移除”，并正确扣减评论数。
- 每次管理员处置记录固定原因、处置时间、目标评论和管理员身份审计信息；普通用户不能读取或写入审计集合。
- 普通用户伪造管理员删除请求必须由云端返回 `FORBIDDEN`，且评论、评论数和审计记录均不得变化。

### 12.5 清除账号数据

用户执行现有“清除所有数据”时：

- 撤回其发布的全部分享手牌，包括广场、全部好友和指定好友范围。
- 撤回所有尚未导入的玩家名片。
- 解除全部好友关系。
- 停止展示其好友统计和排行榜数据。
- 已被其他用户导入的玩家名片副本不被远程删除。
- 管理员处置审计不作为业务数据恢复来源；与被清除账号相关的评论、分享、目标作者或管理员社交 ID 必须在清除流程中脱敏，审计只保留动作、固定原因和时间。

## 13. 交付里程碑

### 13.1 里程碑一：好友关系与资料基础

- 好友邀请、申请、接受、拒绝和解除。
- 好友页与玩家库分区。
- 好友主页和统计可见设置。
- 本周、本月、累计排行榜。
- 玩家名片分享、预览、覆盖或新建导入。
- 消息中心基础能力。

### 13.2 里程碑二：分享手牌互动

- BB 化匿名分享快照。
- 广场、全部好友和指定好友三种范围。
- 统一动态列表与分享手牌详情。
- 文字、Emoji、内置贴纸和一层回复。
- 点赞、取消点赞、撤回和权限实时失效。
- 评论、回复、点赞和定向分享消息。
- 广场基础数据校验和频率限制。

两个里程碑全部通过验收后，一次性发布完整好友系统。

## 14. 可拓展功能

以下方向不进入第一版，但可以在核心好友系统稳定后独立评估：

1. 小程序内账号 ID 搜索。
2. 好友聊天。
3. 广场热门、关注、话题和搜索等内容发现能力。
4. 普通文字动态。
5. 成就和称号主动分享。
6. 微信订阅消息。
7. 更多排行榜维度。
8. 经用户授权的未来手牌自动发布规则。
9. 好友分组及分组可见范围。
10. 玩家名片持续协作和版本更新。
11. 扩展到昵称、图片、好友范围评论等更多内容类型的检测；广场文字评论检测已经属于第一阶段能力。
12. 帖子及评论逐条举报、完整审核工作台和用户申诉；管理员紧急删除评论已经属于第一阶段能力。
13. 用户屏蔽、评论本地隐藏和高级社区治理。
14. 按“广场 / 好友”临时筛选当前动态信息流，但不拆分成独立页面。
15. 玩家名片微信系统分享卡片；必须继续绑定原指定接收人权限，并解决卡片可被转发给无权限用户时的误导问题。

每个拓展方向必须重新评估隐私、内容安全、平台审核和现有数据所有权，不能直接把私有数据表改为跨账号可读。

## 15. 平台限制与产品红线

1. 普通微信小程序不能自动读取微信好友关系。
2. 好友接口不能泄露盈亏、资金曲线和小时盈利。
3. 不能把私有手牌、场次或玩家库文档直接开放给其他账号。
4. 不能在用户未确认快照和可见对象时批量发布历史手牌。
5. 不能因为分享而额外暴露未摊牌的对手底牌。
6. 私人玩家 Note 和 Leak 标签只能通过明确的玩家名片流程分享，不能随手牌自动带出。
7. 前端不得自行推断跨账号访问权限，权限结论必须来自云端。

## 16. 验收标准

### 16.1 导航与入口

1. 底部导航仍显示“场次 / 手牌 / 玩家 / 统计 / 我的”。
2. 玩家页可以在“好友”和“玩家库”之间切换。
3. 好友区包含“动态 / 好友 / 排行榜”；动态为单一信息流，当前可读的三种范围内容按发布时间倒序混排并显示范围标签。
4. 玩家页右上角存在消息入口，底部玩家 tab 能显示未读红点。
5. 我的页存在“隐私与分享”入口。

### 16.2 好友关系

1. 微信邀请卡片和个人二维码都只能发起好友申请。
2. 对方确认前双方不能读取好友数据。
3. 重复申请不会产生重复好友记录。
4. 拒绝或解除后 7 天内不能重新申请。
5. 解除好友后双方立即失去好友资料、排行榜和好友范围内容访问权；公开广场内容继续按广场权限访问。

### 16.3 好友主页和排行榜

1. 好友主页只显示允许的投入数据，不显示任何盈亏字段。
2. 好友详情复用玩家详情结构；本地头像、备注名称、玩家类型、Leak、Note 和对战手牌仅当前用户可见和编辑。
3. 好友详情额外显示云端称号、累计时长和记录手牌数，其中统计字段受对方可见开关控制。
4. 关闭统计可见后，好友主页隐藏时长和手数。
5. 关闭统计可见后，用户退出全部排行榜。
6. 排行榜支持本周、本月和累计，默认本周。
7. 排名只使用有效牌局分钟数，时长相同显示并列。
8. 排行榜显示 Top 10；榜外固定显示自己的实际名次。

### 16.4 分享手牌

1. 默认全部好友可见，可切换为广场或指定好友，三种范围互斥。
2. 广场分享允许已登录的非好友访问；全部好友分享仅好友可访问；指定分享仅被选中的好友可访问。
3. 分享快照不包含真实金额、货币单位、盈亏、地点或玩家库字段。
4. 所有筹码数值统一使用 BB。
5. Hero 显示为 `Hero`，其他玩家使用稳定原创代号。
6. 未摊牌的对手底牌不能进入快照。
7. 原手牌编辑不会静默修改已发布快照。
8. 删除原手牌会自动撤回关联分享。

### 16.5 评论和点赞

1. 当前发布范围内的已登录用户可以发送文字、Emoji 或一张内置贴纸。
2. 回复最多一层。
3. 评论者可以删除自己的评论。
4. 发布者不能删除其他人的评论。
5. 同一用户对同一分享最多一个有效点赞。
6. 解除好友后不能继续互动好友范围和指定范围的分享；广场分享仍按公开权限处理，历史互动保留。

### 16.6 玩家名片

1. 一次只分享一张名片给一位指定好友。
2. 名片只包含头像、名称、玩家类型、Leak 标签和 Note。
3. 名片不包含别名、对战手牌、统计和系统字段。
4. 未导入名片 7 天后失效。
5. 接收方必须预览后手动保存。
6. 同名时只能选择覆盖已有玩家或新建玩家。
7. 覆盖不会删除本地对战手牌关联。
8. 已导入副本不受撤回或解除好友影响。

### 16.7 消息与异常

1. 好友申请、申请通过、定向手牌、评论、回复、点赞和玩家名片产生站内消息。
2. 消息目标失效时显示明确不可访问状态。
3. 社交云端不可用时不伪造成功结果。
4. 社交功能故障不影响现有私有记录功能。
5. 所有跨账号操作由云端鉴权并具备幂等保护。

## 17. 测试用例

### 17.1 权限矩阵

1. 已登录非好友访问广场分享，成功；未登录访问失败。
2. 非好友访问全部好友分享，返回无权限。
3. 好友访问全部好友分享，成功。
4. 好友但未被选中访问指定分享，返回无权限。
5. 被选中好友访问指定分享，成功。
6. 解除好友后使用旧页面和旧令牌访问好友范围或指定范围，返回无权限；广场分享仍可访问。
7. 分享撤回、名片过期或统计关闭后，旧缓存刷新得到最新权限状态。

### 17.2 数据脱敏

1. 对包含 CNY、HKD 或 USD 金额的原手牌生成快照，结果只包含 BB。
2. 原手牌包含真实玩家名称、照片、类型、Leak 标签和 Note 时，快照均不包含这些字段。
3. 同一非 Hero 座位在翻前、翻牌、转牌和河牌保持同一原创代号。
4. 对手未摊牌时，快照不包含对手底牌。
5. 好友主页和排行榜返回体不包含利润、胜率或资金字段。

### 17.3 好友状态

1. 双方同时申请只生成一条关系。
2. 重复点击接受保持已接受状态。
3. 拒绝后 7 天内重新申请被拒绝。
4. 解除后 7 天内重新申请被拒绝。
5. 解除好友后排行榜、主页及好友范围分享的读取和互动权限同时失效；广场内容权限不依赖好友关系。

### 17.4 评论互动

1. 发送 300 字评论成功，301 字失败并保留输入。
2. 发送不存在的贴纸 ID 失败。
3. 评论者删除自己的评论成功。
4. 发布者尝试删除他人评论失败。
5. 重复点赞只存在一个点赞，取消后可以重新点赞。
6. 撤回分享后不能新增评论或点赞。

### 17.5 玩家名片

1. 非指定好友使用转发链接访问失败。
2. 指定好友在 7 天内预览成功，过期后失败。
3. 导入新玩家后生成独立玩家记录。
4. 覆盖同名玩家后五类名片字段更新，对战手牌关联保持不变。
5. 分享者撤回后，尚未导入者无法访问，已经导入者的副本保留。

### 17.6 排行榜

1. 未结束场次不进入排行。
2. 本周、本月边界按北京时间计算。
3. 分钟数相同显示并列名次。
4. 关闭统计可见后从三个范围中全部移除。
5. 排行榜计算和返回体不读取盈亏字段。

### 17.7 回归

1. 场次列表、手牌列表、玩家库、统计和我的页面正常启动。
2. 云端社交请求超时不阻塞新增场次或记录手牌。
3. 现有玩家 Note、Leak 标签和对战手牌数据不因社交模型迁移而变化。
4. 现有账号登录和 `ownerOpenId` 私有数据隔离继续生效。

## 18. 产品指标

- 好友邀请接受率。
- 建立好友后首次发布手牌的比例。
- 分享手牌获得评论或点赞的比例。
- 玩家名片预览后的导入率。
- 周排行榜访问率。
- 用户关闭好友统计可见的比例。
- 分享撤回率和解除好友率。

以上指标只用于验证好友功能价值，不采集或分析用户盈亏。
