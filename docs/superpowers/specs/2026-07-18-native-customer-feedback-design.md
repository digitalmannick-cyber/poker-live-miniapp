# 个人小程序原生客服反馈入口设计

## 背景

当前小程序为个人主体，不能绑定要求非个人主体且主体一致的企业微信“微信客服”。第一版反馈功能改用微信小程序原生客服会话，满足已登录用户通过文字和截图反馈 Bug、使用问题和功能建议的需求。

## 目标

- 在“我的”页面增加清晰的“反馈与建议”入口。
- 仅向已登录小程序账号的用户展示入口。
- 用户点击后直接进入微信小程序原生客服会话。
- 用户在微信客服会话内发送文字和截图，由维护者在微信公众平台客服工具中查看和回复。

## 非目标

第一版不包含以下内容：

- 自定义反馈表单或反馈类型选择器。
- 小程序内图片选择、压缩或上传逻辑。
- `feedbacks` 云数据库集合或反馈云函数。
- 管理员反馈列表、分类、优先级、状态或内部备注。
- 飞书、GitHub Issues 或其他问题清单同步。
- 企业微信注册、认证或“微信客服”接入。

## 用户流程

1. 已登录用户进入“我的”页面。
2. 用户在“帮助与反馈”区域看到“反馈与建议”。
3. 用户点击入口。
4. 微信打开该小程序的原生客服会话。
5. 用户在会话中发送问题说明和截图。
6. 维护者通过微信公众平台客服工具接收、追问和回复。

未登录用户不显示该入口，需先通过现有微信账号登录流程登录小程序。

## 页面设计

### 入口位置

在 `pages/profile/profile` 的设置内容中增加“帮助与反馈”分区。该分区放在偏好类功能之后、危险的数据重置或退出登录操作之前，避免与数据管理操作混淆。

### 入口内容

- 标题：`反馈与建议`
- 说明：`反馈问题、Bug 或功能建议`
- 右侧：沿用“我的”页面现有设置行箭头样式
- 点击区域：整行可点击

入口必须使用微信原生 `<button open-type="contact">`，不能用普通 `<view>` 模拟，否则无法打开客服会话。按钮外观应复用现有 `setting-row`、`setting-main`、`setting-title`、`small muted` 和 `setting-arrow` 视觉语言，并清除微信按钮默认边框、背景和内边距。

## 技术设计

### 页面逻辑

- 不新增页面路由。
- 不新增 JavaScript 提交逻辑。
- 不请求相册、相机或存储权限；图片由微信客服会话自身处理。
- 不向云函数、云数据库或第三方服务写入反馈数据。
- 使用现有 `accountLoggedOut` 页面状态控制入口可见性，条件与其他登录后设置保持一致。

### 微信公众平台配置

上线前，维护者需要在微信公众平台的小程序后台进入“客服消息 / 客服人员”，绑定用于接收反馈的个人微信号。若未配置客服人员，页面代码仍可构建，但无法证明维护者能够正常接收和回复消息。

## 异常与边界

- 客服会话只能在支持该开放能力的微信真机环境中完成最终验证；开发者工具不能作为完整会话验证证据。
- 网络不可用或微信客服能力暂不可用时，由微信客户端展示系统失败状态，第一版不增加自定义重试页面。
- 用户退出小程序账号后入口立即隐藏。
- 原始反馈保留在微信客服会话中，第一版不承诺结构化分类、处理状态或问题追踪。

## 验收标准

1. 已登录用户在“我的”页面能看到“帮助与反馈 / 反馈与建议”。
2. 未登录用户看不到“反馈与建议”。
3. 入口是 `open-type="contact"` 的原生 `button`，点击整行均可触发客服会话。
4. 入口样式与现有设置卡片一致，没有微信按钮默认边框或异常内边距。
5. `app.json` 不新增反馈页面路由。
6. 项目不新增反馈数据库、云函数、图片上传或第三方同步代码。
7. 配置客服人员后，在真机预览中可打开客服会话、发送文字和截图，并由已绑定客服账号接收。

## 测试方案

### 自动化检查

- 增加聚焦回归测试，断言 `profile.wxml` 中存在 `open-type="contact"` 的反馈按钮。
- 断言反馈入口受 `!accountLoggedOut` 控制。
- 断言入口包含既定标题和说明文字。
- 断言反馈按钮样式清除了微信按钮默认伪元素边框。
- 断言 `app.json` 未增加反馈页面。

### 手工验证

1. 在微信公众平台绑定客服人员。
2. 对真实工作区执行微信开发者工具自动预览。
3. 使用已登录账号扫码进入“我的”。
4. 点击“反馈与建议”，确认进入客服会话。
5. 分别发送一条文字和一张截图。
6. 使用已绑定客服账号确认两条消息均可见，并发送回复。
7. 退出小程序账号，确认反馈入口不再显示。
