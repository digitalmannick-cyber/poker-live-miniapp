# 单一进行中 Session 与复盘状态筛选设计

## 目标

限制同一时间只能存在一个进行中的 Session，并将复盘页不断增长的具体牌局筛选改为稳定的牌局状态筛选。

## 单一进行中 Session

### 页面行为

- Session 列表存在进行中 Session 时，“新建”按钮显示禁用态。
- 用户点击禁用态按钮时统一提示：“创建新 Session 前需先结束当前 Session”。
- 记牌页的创建 Session 入口使用相同判断和提示。
- 直接进入 Session 创建页时也执行检查；发现进行中 Session 后提示并返回，避免展示一个最终无法保存的表单。
- 没有进行中 Session 时，各入口维持现有创建流程。

### 数据层兜底

- `dataService.createSession()` 在写入前查询当前 Session 列表。
- 如果已经存在 `status === 'active'` 的 Session，则抛出带稳定错误码的业务错误，不创建任何记录，也不触发云同步。
- 本地 store 和云仓库的创建方法保留同样的防御性校验，防止未来新增入口绕过服务层。
- Session 创建页捕获该业务错误并显示统一提示，其他错误继续显示创建失败提示。

## 复盘状态筛选

### 交互

- 筛选弹窗中的“牌局”改名为“牌局状态”。
- 只显示两个选项：“进行中”和“已结束”，不再渲染具体 Session 列表，也不提供“全部牌局”。
- 初次进入复盘页时：存在进行中 Session 则默认选中“进行中”；不存在则默认选中“已结束”。
- 用户主动选择“已结束”后，只要该状态仍有意义，后续刷新保持用户选择。
- 当前选择为“进行中”但刷新时已不存在进行中 Session，则自动切换到“已结束”。
- 重置筛选时恢复上述动态默认状态。
- 筛选摘要显示“进行中牌局”或“已结束牌局”。

### 数据流

- 前端筛选字段从 `sessionId` 改为 `sessionStatus`，值只允许 `active` 或 `finished`。
- `getReviewData(filters)` 先取得 Session 列表，再按 `sessionStatus` 得到允许的 Session ID 集合，最后过滤手牌并计算摘要。
- 日期、输赢、标签及排序继续沿用现有逻辑，并与状态筛选组合生效。
- 为兼容旧的待应用筛选数据，如果只收到旧 `sessionId`，复盘页根据对应 Session 推导其状态；无法推导时使用动态默认状态。
- 记牌页保存手牌后写入待应用筛选时改为 `sessionStatus: 'active'`。

## 空状态

- 选择“进行中”但进行中 Session 暂无手牌时，列表显示现有“还没有可复盘的手牌”空状态。
- 完全没有 Session 时默认选择“已结束”，同样显示现有空状态，不新增第三种筛选选项。

## 测试范围

- Session 列表和记牌页存在进行中 Session 时均阻止进入创建流程，并显示完全一致的提示。
- 直接访问创建页会被拦截。
- 服务层、本地 store 和云仓库均拒绝创建第二个进行中 Session。
- 没有进行中 Session 时仍可正常创建。
- 复盘筛选只渲染“进行中 / 已结束”。
- 默认状态、无进行中 Session 时的自动切换、刷新保持用户选择和重置逻辑正确。
- 状态筛选与日期、输赢、标签及排序组合后结果正确。
- 旧 `sessionId` 待筛选数据能迁移为正确状态。
- 现有复盘编辑、AI 建议、左滑操作及 Session 总结流程不受影响。

## 非目标

- 不自动结束当前 Session。
- 不新增“全部状态”选项。
- 不改变已结束 Session 的编辑和删除能力。
- 不改变 Session 列表现有排序方式。
